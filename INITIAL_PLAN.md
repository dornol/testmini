# QA Management System (Internal TMS)

## 1. Overview

Internal QA management system for handling test cases and execution
results per project.

Goals:

- Project isolation
- Test case versioning
- Release-based test runs
- Detailed failure tracking
- File attachments
- Multi-user collaboration
- Conflict prevention (Soft Lock + Optimistic Lock)
- Real-time execution sync (WebSocket)
- Multi-language support (ko / en)

Automation integration will be introduced in Phase 2.

## 4. Authentication

### Provider: better-auth

Already configured with:

- `betterAuth` with drizzle adapter (`provider: 'pg'`)
- `emailAndPassword: { enabled: true }`
- `sveltekitCookies` plugin for SvelteKit integration
- Session available via `event.locals.user` / `event.locals.session`

### Login Types

- Local account (email / password) ✅
- OIDC/OAuth2 providers — 커스텀 OAuth/PKCE 핸들러 (런타임 IdP 관리) ✅

### Session Strategy

- httpOnly cookie (better-auth default)
- DB-based session storage (better-auth manages `session` table)
- No Redis dependency for sessions

### Auth Schema

Generated by `better-auth generate` command:

```bash
pnpm auth:schema
```

Tables managed by better-auth:

- `user` — id, email, name, emailVerified, image, createdAt, updatedAt
- `session` — id, userId, expiresAt, token, ipAddress, userAgent
- `account` — id, userId, providerId, providerAccountId, ...
- `verification` — id, identifier, value, expiresAt

> Custom auth tables (User, UserAuth) are NOT needed.
> better-auth handles all auth-related schema automatically.

### TestCaseVersion

| Column          | Type      | Description                |
| --------------- | --------- | -------------------------- |
| id              | serial    | PK                         |
| test_case_id    | integer   | FK → TestCase              |
| version_no      | integer   | Sequential version number  |
| title           | text      | Test case title            |
| precondition    | text      | Prerequisites              |
| steps           | jsonb     | Ordered test steps (below) |
| expected_result | text      | Overall expected result    |
| priority        | enum      | LOW / MEDIUM / HIGH / CRITICAL |
| revision        | integer   | Optimistic lock counter    |
| updated_by      | text      | FK → user                  |
| created_at      | timestamp | Version creation timestamp |

**Steps JSONB Schema:**

```typescript
type TestStep = {
  order: number;       // Step sequence (1, 2, 3...)
  action: string;      // Action to perform
  expected: string;    // Expected result for this step
};
// Column type: TestStep[]
```

New row is created on each update. `revision` field is used for
optimistic locking within the same version.

2. API saves file to local storage and metadata to `attachment` table
3. Download via API endpoint with streaming response

## 7. Concurrency Strategy

### TestCase Editing ✅

**Soft Lock (Redis):**

- Key: `lock:testcase:{testCaseVersionId}`
- Value: `userId`
- TTL: 10 minutes

Flow:

1. Create lock when editing starts
2. Deny editing if lock exists (show lock holder info)
3. Release on save / cancel
4. Auto-expire by TTL

**Optimistic Lock ✅:**

```sql
UPDATE test_case_version
SET ..., revision = revision + 1
WHERE id = ? AND revision = ?
```

If 0 rows affected → conflict detected → notify user.

### TestRun Execution Screen ✅

1. Save execution result
2. Broadcast SSE event via Redis Pub/Sub
3. Update only modified row on other clients
4. Use optimistic lock for integrity

## 11. Performance Considerations

- Mandatory server-side pagination (cursor or offset)
- Database indexes:
  - `test_case(project_id)`
  - `test_case_version(test_case_id, version_no DESC)`
  - `test_execution(test_run_id, status)`
  - `attachment(reference_type, reference_id)`
- Virtual scrolling for run execution screen
- GIN index on `steps` JSONB if search is needed

## 12. Testing Strategy

### Unit Tests (Vitest, node environment)

- Server-side logic, utilities, validation
- File pattern: `src/**/*.{test,spec}.{js,ts}`

### Component Tests (Vitest + Playwright)

- Svelte component rendering and interaction
- File pattern: `src/**/*.svelte.{test,spec}.{js,ts}`
- Runs in headless Chromium

### Configuration

- `requireAssertions: true` — every test must have at least one assertion
- Server tests excluded from browser runner
- Component tests excluded from server runner
